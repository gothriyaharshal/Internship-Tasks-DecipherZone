Important part.

Your doubt is very good. The confusion is because of final variable rules vs constructor initialization.

You said:

if any field is final then we have to provide data at that time ...not at object creation

Actually the rule is:

A final instance variable must be assigned exactly once before the constructor finishes.
It does not mean you must initialize it at declaration time.

There are multiple ways.

1. Initialize at declaration
class Test
{
    final int x = 10;
}

Here value is given immediately.

2. Initialize inside constructor (most common)
class Test
{
    final int x;

    Test(int value)
    {
        x = value;
    }
}

Now:

Test t = new Test(50);

During object creation:

new Test(50)
       |
       |
 constructor runs
       |
       |
x = 50

After constructor completes:

x cannot change

Because reference is same.

Memory:

Before:

arr
 |
 v
[0,0,0]

After:

arr
 |
 v
[100,0,0]

Reference did not change.

But this is not allowed:

because reference changes.

Now String becomes immutable because of extra design:

private final byte[] value;

AND

String never gives this array outside.

Example:

You cannot do:

because:

value is private
no method returns the original array
methods create new String objects

does not do:

value = old array + new chars

It does:

new String(new array)

So the exact chain is:

final
   |
   prevents reference change

private
   |
   prevents outside access

String methods
   |
   never modify array

new object created
   |
   immutability achieved

final alone does not make String immutable. It is only one part of the design.
